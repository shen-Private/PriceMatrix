# PriceMatrix 資料庫設計補充說明

## 🎯 核心問題

```
客戶數: 1,000
商品數: 1,600 (預計)
理論最大折扣紀錄: 1,000 × 1,600 = 1,600,000 筆

問題:
要怎麼處理這麼多筆資料?
```

---

## ❌ 錯誤做法: 預先建立所有紀錄

### 天真的想法
```sql
-- 錯誤示範:建立所有可能的組合
INSERT INTO discounts (customer_id, product_id, discount_ratio)
SELECT 
  c.customer_id, 
  p.product_id, 
  1.0 AS discount_ratio  -- 預設原價
FROM customers c
CROSS JOIN products p;

-- 結果:直接插入 1,600,000 筆紀錄 😱
```

### 問題分析
```
1. 空間浪費
   └─ 160 萬筆 × 每筆約 50 bytes = 80 MB
   └─ 加上索引可能膨脹到 200+ MB
   └─ 大部分紀錄都是「預設值 1.0」(無意義)

2. 查詢效能差
   └─ SELECT 時要掃描 160 萬筆
   └─ 即使有索引,資料量太大也會慢

3. 維護困難
   └─ 新增一個商品 → 要插入 1000 筆折扣紀錄
   └─ 新增一個客戶 → 要插入 1600 筆折扣紀錄
   └─ 刪除商品/客戶 → 要刪除對應的所有折扣紀錄

4. 浪費記憶體
   └─ MySQL 的 Buffer Pool 會被無用資料佔滿
```

---

## ✅ 正確做法: 稀疏矩陣設計 (Sparse Matrix)

### 核心概念

```
只儲存「有折扣」的紀錄
沒紀錄 = 使用預設折扣 (1.0 = 原價)
```

### 實際情境分析

```
客戶 A:
├─ 只買過 10 個商品
└─ 只建立 10 筆折扣紀錄

客戶 B (VIP):
├─ 買過 200 個商品
└─ 建立 200 筆折扣紀錄

客戶 C (新客戶):
├─ 還沒買過任何商品
└─ 0 筆折扣紀錄

總計:
├─ 1000 個客戶
├─ 平均每個客戶買過 50-100 個商品
└─ 實際折扣紀錄: 50,000 ~ 100,000 筆
   (節省 90% 空間!)
```

## 📐 資料表設計

### Discounts 表 (折扣表)

```sql
CREATE TABLE discounts (
  discount_id INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  product_id INT NOT NULL,
  discount_ratio DECIMAL(5,4) NOT NULL DEFAULT 1.0000,
  -- 折扣率: 0.8500 = 85折
  -- 使用 DECIMAL 確保精確計算
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  updated_by VARCHAR(50),
  -- 誰最後修改的 (未來權限系統會用到)
  
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id),
  FOREIGN KEY (product_id) REFERENCES products(product_id),
  
  UNIQUE KEY uk_customer_product (customer_id, product_id)
  -- 防止同一個客戶對同一個商品有多筆折扣紀錄
);

-- 重要索引
CREATE INDEX idx_customer ON discounts(customer_id);
CREATE INDEX idx_product ON discounts(product_id);
```

### Discount_Change_Logs 表 (折扣修改紀錄)

```sql
CREATE TABLE discount_change_logs (
  log_id INT PRIMARY KEY AUTO_INCREMENT,
  discount_id INT,
  -- 如果是刪除折扣,這個 FK 可以是 NULL
  
  customer_id INT NOT NULL,
  product_id INT NOT NULL,
  -- 冗餘儲存 customer_id 和 product_id
  -- 這樣即使 discount 被刪除,log 還能查到是誰的折扣
  
  old_ratio DECIMAL(5,4),
  -- 修改前的折扣率 (新增時為 NULL)
  
  new_ratio DECIMAL(5,4),
  -- 修改後的折扣率 (刪除時為 NULL)
  
  change_type ENUM('CREATE', 'UPDATE', 'DELETE'),
  -- 修改類型
  
  changed_by VARCHAR(50) NOT NULL,
  -- 誰改的
  
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  -- 什麼時候改的
  
  reason VARCHAR(200),
  -- 改的原因 (選填)
  
  FOREIGN KEY (discount_id) REFERENCES discounts(discount_id) ON DELETE SET NULL,
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id),
  FOREIGN KEY (product_id) REFERENCES products(product_id)
);

-- 索引 (用於快速查詢歷史)
CREATE INDEX idx_customer_product ON discount_change_logs(customer_id, product_id);
CREATE INDEX idx_changed_at ON discount_change_logs(changed_at);
CREATE INDEX idx_changed_by ON discount_change_logs(changed_by);
```

## 🔍 查詢邏輯設計

### 場景 1: 查詢客戶的所有商品折扣

**需求:**
業務要看「客戶 A 對所有商品的折扣」

**錯誤做法:**
```sql
-- ❌ 這樣會漏掉「沒折扣紀錄」的商品
SELECT 
  p.product_name,
  d.discount_ratio
FROM products p
LEFT JOIN discounts d ON p.product_id = d.product_id
WHERE d.customer_id = 123;
-- 問題:沒折扣紀錄的商品不會出現
```

**正確做法:**
```sql
-- ✅ 用 LEFT JOIN + COALESCE 給預設值
SELECT 
  p.product_id,
  p.product_name,
  p.base_price,
  COALESCE(d.discount_ratio, 1.0000) AS discount_ratio,
  -- 如果沒折扣紀錄,用預設值 1.0
  p.base_price * COALESCE(d.discount_ratio, 1.0000) AS final_price
FROM products p
LEFT JOIN discounts d 
  ON p.product_id = d.product_id 
  AND d.customer_id = 123
WHERE p.category_id = 5;  -- 可選:只看某分類
```

**查詢說明:**
```
LEFT JOIN:
└─ 即使商品沒有折扣紀錄,也會出現在結果中

COALESCE(d.discount_ratio, 1.0000):
└─ 如果 d.discount_ratio 是 NULL (沒紀錄)
└─ 就用 1.0000 (原價)

效能:
└─ 只掃描需要的商品 (可用 category_id 過濾)
└─ 不需要掃描 160 萬筆折扣紀錄
```

### 場景 2: 查詢某商品對所有客戶的折扣

**需求:**
管理員要看「商品 X 對所有客戶的折扣分佈」

```sql
-- ✅ 只查有折扣的客戶
SELECT 
  c.customer_id,
  c.customer_name,
  d.discount_ratio,
  p.base_price * d.discount_ratio AS final_price
FROM discounts d
JOIN customers c ON d.customer_id = c.customer_id
JOIN products p ON d.product_id = p.product_id
WHERE d.product_id = 456
ORDER BY d.discount_ratio ASC;
-- 按折扣率排序,看誰的折扣最優惠
```

**查詢說明:**
```
為什麼用 JOIN 而不是 LEFT JOIN?
└─ 因為只想看「有買過的客戶」
└─ 沒買過的客戶沒有意義

效能:
└─ 只掃描有折扣的紀錄 (可能只有幾千筆)
└─ 用 product_id 索引快速定位
```

### 場景 3: 批次修正折扣

**需求:**
「給所有 VIP 客戶的 A 分類商品打 8 折」

**步驟 1: 找出目標客戶與商品**
```sql
-- 先找出符合條件的組合
SELECT 
  c.customer_id,
  p.product_id
FROM customers c
CROSS JOIN products p
WHERE c.is_vip = 1  -- VIP 客戶
  AND p.category_id = 1;  -- A 分類商品
-- 假設結果有 500 筆組合
```

**步驟 2: 插入或更新折扣**
```sql
-- MySQL 的 INSERT ... ON DUPLICATE KEY UPDATE
INSERT INTO discounts (customer_id, product_id, discount_ratio, updated_by)
VALUES 
  (101, 201, 0.8000, 'admin'),
  (101, 202, 0.8000, 'admin'),
  (102, 201, 0.8000, 'admin'),
  -- ... 500 筆
ON DUPLICATE KEY UPDATE
  discount_ratio = VALUES(discount_ratio),
  updated_by = VALUES(updated_by),
  updated_at = CURRENT_TIMESTAMP;
```

**步驟 3: 記錄到 Log**
```sql
-- 在 Java Service 層執行
-- 每次修改都插入一筆 log
INSERT INTO discount_change_logs 
  (discount_id, customer_id, product_id, old_ratio, new_ratio, change_type, changed_by, reason)
VALUES 
  (1234, 101, 201, 1.0000, 0.8000, 'UPDATE', 'admin', 'VIP 促銷活動');
```

## 📊 效能優化策略

### 1. 索引設計

```sql
-- 複合索引 (最重要!)
UNIQUE KEY uk_customer_product (customer_id, product_id)
-- 用途:
-- 1. 防止重複
-- 2. 快速查詢「某客戶對某商品」的折扣

-- 單欄索引
INDEX idx_customer (customer_id)
-- 用途:查詢「客戶的所有折扣」

INDEX idx_product (product_id)
-- 用途:查詢「商品的所有折扣」
```

### 2. 查詢優化技巧

**使用 JOIN 而不是子查詢**
```sql
-- ❌ 慢
SELECT 
  (SELECT customer_name FROM customers WHERE customer_id = d.customer_id),
  (SELECT product_name FROM products WHERE product_id = d.product_id),
  discount_ratio
FROM discounts d;

-- ✅ 快
SELECT 
  c.customer_name,
  p.product_name,
  d.discount_ratio
FROM discounts d
JOIN customers c ON d.customer_id = c.customer_id
JOIN products p ON d.product_id = p.product_id;
```

**使用 LIMIT 限制結果**
```sql
-- 業務只需要看前 100 筆,不要全部撈出來
SELECT * FROM discounts 
WHERE customer_id = 123
LIMIT 100;
```

### 3. 批次操作優化

**使用事務 (Transaction)**
```java
// Java Service 層
@Transactional
public void batchUpdateDiscounts(List<DiscountDTO> discounts) {
    for (DiscountDTO dto : discounts) {
        discountMapper.updateDiscount(dto);
        logMapper.insertLog(dto);  // 記錄到 Log
    }
    // 如果中間有錯誤,全部回滾
}
```

**批次插入 (Batch Insert)**
```java
// 一次插入多筆,減少資料庫往返次數
discountMapper.batchInsert(discountList);
// MyBatis 會自動組成:
// INSERT INTO discounts VALUES (...), (...), (...)
```

## 🎯 設計原則總結

### 1. 稀疏矩陣 (Sparse Matrix)
```
只儲存「有意義」的資料
└─ 有折扣才建紀錄
└─ 沒紀錄 = 預設值
```

### 2. 冗餘與效能的平衡
```
適度冗餘:
└─ Log 表冗餘儲存 customer_id, product_id
└─ 即使主表被刪,Log 還能查到

不要過度冗餘:
└─ 不要在 discounts 表儲存 customer_name
└─ 用 JOIN 查詢,保持資料一致性
```

### 3. 索引策略
```
經常查詢的欄位 → 建索引
經常一起查詢的欄位 → 建複合索引
很少查詢的欄位 → 不要建索引 (浪費空間)
```

### 4. Log 系統完整性
```
記錄所有修改:
├─ CREATE (新增折扣)
├─ UPDATE (修正折扣)
└─ DELETE (刪除折扣)

記錄完整資訊:
├─ 誰改的 (changed_by)
├─ 什麼時候改的 (changed_at)
├─ 改了什麼 (old_ratio → new_ratio)
└─ 為什麼改 (reason, 選填)
```

## 💡 實際數字估算

### 空間需求

```
方案 A (錯誤):預先建立所有紀錄
├─ 紀錄數: 1,600,000 筆
├─ 每筆大小: 約 50 bytes
├─ 索引大小: 約 100 bytes
└─ 總空間: 1,600,000 × 150 bytes ≈ 240 MB

方案 B (正確):稀疏矩陣
├─ 紀錄數: 50,000 ~ 100,000 筆
├─ 每筆大小: 約 50 bytes
├─ 索引大小: 約 100 bytes
└─ 總空間: 100,000 × 150 bytes ≈ 15 MB

節省空間: 240 MB - 15 MB = 225 MB (節省 94%)
```

### 查詢效能

```
方案 A:
└─ SELECT ... WHERE customer_id = 123
└─ 需要掃描 1,600 筆 (該客戶的所有商品)
└─ 即使有索引,資料量大

方案 B:
└─ SELECT ... WHERE customer_id = 123
└─ 只掃描 50 ~ 100 筆 (該客戶實際買過的商品)
└─ 快 10-20 倍!
```

## 🚀 未來擴充考量

### 如果商品數量暴增到 10,000 個?

```
理論最大紀錄: 1,000 × 10,000 = 10,000,000 筆

實際紀錄 (稀疏矩陣):
└─ 假設每個客戶平均買 100 個商品
└─ 1,000 × 100 = 100,000 筆
└─ 仍然很小!

結論:
└─ 稀疏矩陣設計有極強的擴充性
└─ 即使商品數量 × 10,也不會爆炸
```

### 如果需要「歷史折扣」查詢?

```
目前設計:
└─ 只保留「當前折扣」
└─ Log 表只記錄「修改歷史」

未來可擴充:
└─ 在 discounts 表增加 effective_date
└─ 同一個客戶×商品可以有多筆折扣 (不同時間)
└─ 查詢時加上時間條件
```

## 📝 總結

### 核心設計理念

```
1. 只儲存必要的資料
   └─ 有折扣才建紀錄

2. 用查詢邏輯補齊預設值
   └─ COALESCE(discount_ratio, 1.0)

3. 用索引保證效能
   └─ 複合索引 (customer_id, product_id)

4. 用 Log 確保可追溯
   └─ 所有修改都有紀錄
```

### 這樣設計的好處

```
✅ 空間效率: 節省 90% 以上
✅ 查詢效率: 快 10-20 倍
✅ 維護容易: 新增商品/客戶不需要批次建折扣
✅ 擴充性強: 商品數量增加 10 倍也不怕
✅ 可追溯性: 完整的修改歷史
```
