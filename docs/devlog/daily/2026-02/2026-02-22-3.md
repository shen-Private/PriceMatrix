## 踩過的坑

**import 沒加（多次）**
Java 所有用到的 class 都要明確 import，`InventoryStock`、`OutsourceInquiryLog`、`Optional` 都因為沒加 import 而報 `cannot find symbol`。

**Repository 變數名稱不一致**
`InventoryItemService` 注入的變數叫 `itemRepository`，但 `getOverview()` 裡寫成 `inventoryItemRepository`。Java 靠變數名稱找東西，名稱對不上就報錯。

**方法貼錯 Repository**
`findByItemId()` 和 `findTopByItemId...()` 貼進了錯誤的 Repository，導致找不到對應 class。每個 Repository 只管自己的 Entity。

**`@GetMapping` 路徑重複**
`@RequestMapping("/api/inventory/items")` + `@GetMapping("/items/overview")` = `/api/inventory/items/items/overview`（多了一層）。改成 `@GetMapping("/overview")` 解決。

**前端 `inventory` 資料夾放錯位置**
建在 `modules/pricing/inventory/` 而不是 `modules/inventory/`，導致 `Module not found`。從 VS Code 拖移修正。

**React key prop 警告**
`filteredItems.map()` 回傳多個 `<tr>` 時用 `<>`（Fragment 簡寫），簡寫版不能加 key。改為 `<React.Fragment key={item.id}>` 解決。

**Lazy Loading 導致 GET 歷史 500 錯誤**
`OutsourceInquiryLog` 的 `item` 欄位用 `FetchType.LAZY`，Jackson 轉 JSON 時碰到 Hibernate Proxy 報錯。加上 `@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})` 解決。

**Repository 方法名稱需用底線**
`findByItemIdOrderByConfirmedAtDesc` 對應的是 `itemId` 欄位，但 Entity 裡是 `item`（物件）不是 `itemId`（數字）。JPA 需要寫成 `findByItem_IdOrderByConfirmedAtDesc` 才能正確對應到關聯物件的 id。

## 學到的概念

**創新端點的順序：由內而外**
```
Entity → Repository → Service → Controller
```
每一層依賴前一層，從內往外建不會遇到「找不到 class」的問題。

**查錯誤的順序：由外而內**
```
錯誤訊息 → Controller → Service → Repository → Entity → DB
```
編譯錯誤（cannot find symbol）→ 找 import、變數名、路徑
執行錯誤（500）→ 看 IntelliJ console log 的行號

**DTO（Data Transfer Object）**
「專門用來傳資料的物件」。現在 Controller 直接回傳 Entity，Entity 同時兼職DB對應和JSON序列化。DTO 可以把這兩件事分開，Entity 只管DB，DTO 只管給前端。目前用 `@JsonIgnoreProperties` 暫解，之後重構。

**Proxy（代理物件）**
Hibernate Lazy Loading 的技巧。用到關聯欄位才去DB撈，沒用到就給一個假的 Proxy 物件佔位。Jackson 不認識 Proxy，所以要用 `@JsonIgnoreProperties` 讓它跳過。

**JPA 關聯欄位查詢語法**
Entity 裡是物件關聯（`private InventoryItem item`），Repository 方法名要用底線穿透：
`findByItem_IdOrderByConfirmedAtDesc` → 對應 `item.id`
