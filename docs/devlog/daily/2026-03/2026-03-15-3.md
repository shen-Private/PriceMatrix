## 設計決策

### 未知條碼の処理方針
- 方案A（現場暫建、事後補齊）を採用
- 倉庫：名称＋廠商のみ入力して暫時建檔（`status: pending`）
- CS：定価・分類・庫存形態・単位を補齊して正式建檔
- 待確認商品は `IN`（入庫）のみ可能（`OUT`・報価は `active` になってから）

### 定価の入力責任
- CS または管理者が入力する（倉庫は入力しない）
- 定価は報価の基準になるため、正確な入力が必要

## 踩到的坑

### `updateProduct` で name が null になる
**症狀：** `PUT /api/products/{id}` → 500 `not-null property references a null`

**原因：** `@RequestBody Product` で受け取っていたため、`categoryId` が `category` オブジェクトとして解釈されず null に。`name` も送っていなかったため null エラー。

**解法：** `UpdateProductRequest` DTO を作成し、`name` / `basePrice` / `categoryId` を個別に受け取るよう変更。

### handleConfirm で商品名が更新されない
**症狀：** CS が商品名を変更しても反映されない。

**原因：** `POST /api/inventory/items` の `productName` に `pendingProducts.find(...).name`（元の名前）を使っていた。

**解法：** `pendingForm.name` に変更。
