## 3. 客戶管理頁面設計

### 3.1 多事務所處理（parent_id 方案）

同一公司旗下多個事務所各自是一筆 customer，透過 parent_id 指向上層公司。

| 情境 | 做法 |
|------|------|
| 獨立客戶 | parent_id = NULL |
| 山田介護（總公司） | parent_id = NULL（上層） |
| 山田介護 渋谷事務所 | parent_id = 山田介護 id |
| 山田介護 新宿事務所 | parent_id = 山田介護 id |

### 3.2 活動時間軸（sales_contact_log）

客戶頁面內顯示該客戶所有相關的 sales_contact_log，依時間排列。

```
1/1   拜訪        客人對OO產品有興趣
2/1   報價        產出報價單 [Q-2026-001]（可點擊）
2/10  電話        價格太高，婉拒
5/10  推薦新商品  推薦XX產品，跟進中
```

### 3.3 成交流程（方案B：手動確認）

業務按「成交」→ 跳出表單補齊正式客戶資料（電話、郵件等）→ 建立後自動關聯 `prospect.customer_id`。

## 4. 日報填寫頁面設計

### 4.1 填寫欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| 日期 | 日期選擇 | 哪天的日報 |
| 客戶 | 關聯選擇 | 關聯到哪個客戶（可多筆） |
| 互動類型 | 下拉選單 | 拜訪 / 電話 / 郵件 / 報價 / 其他 |
| 內容 | 自由文字 | 當天發生的事（無格式限制） |
| 結果 | 下拉選單 | 進展中 / 正面 / 負面 / 待跟進 |
| 下次行動 | 自由文字 | AI 分析用，例：下週再打電話 |

> **填寫介面提示文字：** 建議寫「客戶說了什麼、你說了什麼、下一步是什麼」，不強制但引導業務往有利 AI 分析的方向填寫。

### 4.2 填寫時機

每天結束後統一填寫（非即時填寫）。

> ⚠ 此功能列為 Backlog，待 sales 模組基本完成後再實作。

### 5.1 搜尋對象

Header 上的全域搜尋列，同時搜尋：客戶公司名稱、聯絡窗口人名、Email、電話、報價單編號、商品名稱。

### 5.2 UI 行為

| 元素 | 說明 |
|------|------|
| 輸入框下方 | 即時候補清單（autocomplete） |
| 輸入框上方 | 篩選 Tab：人名 / 公司名稱 / 報價單編號 / Email / 電話 |
| 詳細檢索頁面 | 點擊後跳到獨立頁面，可自由組合多條件查詢 |
