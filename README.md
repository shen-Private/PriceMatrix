# PriceMatrix 專案概述

## 🎯 專案目標

**讓業務人員能快速查詢並修正每個客戶對每個商品的專屬折扣**

---

## 📖 背景故事

### GAS 1.0 的演進歷史

#### 史前時代: Excel 地獄
```
工具: Microsoft Excel
痛點:
├─ 大量函數導致每次開啟都有很高機率當機
├─ 為了輸出格式統一,流程過於複雜
├─ 作業已變成「屬人化」
└─ 每次教其他人使用時,心靈負擔極大
```

#### GAS 1.0: 試算表 + Apps Script
```
工具: Google Sheets + Google Apps Script
設計:
├─ 將試算表當作 DB
├─ 使用 GAS 建立網頁 APP
└─ 目標: 讓業務能在網頁介面操作

實際運作:
└─ 業務習慣「手打計算機算完後直接更新」
   └─ 使用習慣與系統設計有落差

遇到的問題:
├─ 列數過於龐大
├─ 試算表復原時的預覽功能會直接當掉
└─ 系統只做到一半,功能不完整
```

### 為什麼要做 2.0?

#### GAS 1.0 的瓶頸
```
1. 效能問題
   └─ Google Sheets 無法處理大量數據

2. 穩定性問題
   └─ 預覽、復原等功能容易崩潰

3. 擴充性問題
   └─ 難以新增複雜功能(批次修正、log 追蹤等)
```

#### PriceMatrix 2.0 的改善目標
```
1. 真正的資料庫
   └─ 使用 MySQL 取代試算表
   └─ 能處理百萬級資料

2. 企業級架構
   └─ 前後端分離
   └─ 模組化設計,易於擴充

3. 完整功能
   ├─ 單一修正
   ├─ 批次修正
   ├─ 分類批次修正
   └─ Log 追蹤系統

4. 學習目標
   └─ 順便學習 Java + Spring Boot + React
   └─ 打造可展示的作品集
```

## 🎬 典型使用場景

### 場景 1: 業務查詢折扣 (GAS 1.0 的做法)
```
1. 客戶打電話問「我上次買 A 商品的折扣是多少?」
2. 業務打開 Google Sheets
3. 在龐大的試算表中找到客戶那一列
4. 找到商品那一欄
5. 看到折扣率,回覆客戶

痛點:
└─ 試算表太大,搜尋很慢
```

### 場景 2: 業務修正折扣 (GAS 1.0 的做法)
```
1. 業務需要修改某客戶的折扣
2. 打開試算表
3. 手動輸入新的折扣率
4. 完成

痛點:
├─ 沒有紀錄「誰改的」
├─ 沒有紀錄「為什麼改」
└─ 改錯了很難復原
```

### 場景 3: 主管要求批次調整 (GAS 1.0 做不到)
```
主管說:「給所有 VIP 客戶的 A 分類商品打 8 折」

GAS 1.0:
└─ 只能一個一個手動改 (崩潰)

PriceMatrix 2.0 目標:
1. 篩選出所有 VIP 客戶
2. 選擇 A 分類商品
3. 批次修改折扣率為 0.8
4. 系統自動記錄修改歷史
```

### 場景 4: 出問題要查紀錄 (GAS 1.0 做不到)
```
老闆問:「為什麼這個客戶的折扣變了?」

GAS 1.0:
└─ 無法追蹤,只能看試算表的版本歷史(很難用)

PriceMatrix 2.0 目標:
1. 查詢 audit log
2. 看到「誰」在「什麼時候」改成「多少」
3. 看到修改原因(如果有填)
```

## 📊 資料規模

### 當前規模
```
客戶數量: 1,000
商品數量: 600 (預計擴充到 1,600)
折扣紀錄: 理論上最多 1,000 × 1,600 = 1,600,000 筆
同時在線: 預計 3 人,為擴充性設計為 10 人
```

### 實際折扣紀錄估算
```
理論最大值: 1,600,000 筆
  └─ 如果每個客戶都買過每個商品

實際情況:
  └─ 大部分客戶只買少數商品
  └─ 預估實際折扣紀錄: 50,000 ~ 100,000 筆

設計策略:
  └─ 「有折扣才建紀錄」
  └─ 沒紀錄 = 使用預設折扣(1.0 = 原價)
```

### 效能考量
```
查詢頻率: 低 (業務按需查詢)
修正頻率: 低 (偶爾調整折扣)
批次修正: 偶爾 (主管要求時)

效能目標:
├─ 單一客戶查詢: < 1 秒
├─ 單一修正: < 0.5 秒
└─ 批次修正 100 筆: < 5 秒
```

## 🎯 核心功能清單

### MVP (第一版必須有的)
```
□ 查詢客戶的所有折扣
  └─ 輸入客戶名稱,顯示所有商品的折扣

□ 修正單一折扣
  └─ 修改某個客戶對某個商品的折扣率

□ 基本的紀錄追蹤
  └─ 記錄「誰」在「什麼時候」改了「什麼」
```

### 第二階段功能
```
□ 批次修正折扣
  └─ 一次修改多個客戶或多個商品

□ 同分類複數客戶的批次修正
  └─ 「給所有 VIP 客戶的 A 分類商品打 8 折」

□ Log 查詢介面
  └─ 查詢修改歷史,支援復原
```

### 未來可能的功能 (Backlog)
```
□ 統計報表
  └─ 「哪些商品最常被調整折扣?」
  └─ 「哪些客戶的折扣最優惠?」

□ 折扣紀錄分析
  └─ 分析「折扣修改的原因是什麼?」
  └─ 抓出「商品折扣的上限與下限」
  └─ 幫助業務判斷合理的折扣範圍

□ 權限管理系統
  └─ 業務只能改自己負責的客戶
  └─ 主管可以審核折扣修改

□ 智能建議 (如果想玩 AI)
  └─ 根據歷史數據建議「合理的折扣」
```

```
1. 企業級標準
```

### 為什麼選 React + TypeScript?
```
1. 業界主流
   └─ 求職市場需求大

2. 組件化思維
   └─ 適合複雜的 UI 設計

3. TypeScript 強型別
   └─ 與後端 Java 思維一致
```

```
   └─ 能處理百萬級資料

3. 學習資源豐富
   └─ 遇到問題容易找到解答
```

## 📐 架構設計原則

### 前後端分離
```
前端 (React + TS):
└─ 只負責 UI 顯示與使用者互動

後端 (Spring Boot):
└─ 負責業務邏輯與資料驗證

資料庫 (MySQL):
└─ 負責資料持久化
```

### 防禦性設計
```
1. 雙重驗證
   ├─ 前端: UX 層級的即時回饋
   └─ 後端: 業務邏輯的最終防線

2. 不信任前端
   └─ 所有資料都要在後端重新驗證

3. 精確計算
   └─ 使用 BigDecimal 處理金額

4. 併發安全
   ├─ Transaction 事務管理
   └─ Unique Index 防重複
```

### 可擴充性
```
1. 權限系統預留
   └─ 資料庫欄位先挖好洞

2. Log 系統完整
   └─ 未來可以做各種分析

3. 模組化設計
   └─ 新功能可以獨立開發
```
