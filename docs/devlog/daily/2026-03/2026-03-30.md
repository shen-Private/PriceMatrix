  port: 3000,

    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true,
    },
  },
},

## 概念學習

### 錯誤訊息判讀
- `root cause` 之後才是真正原因
- `Failing row contains` 直接印出問題資料
- Entity 層錯誤（Hibernate 擋）vs DB 層錯誤（DB 擋）兩層分開處理

### 無限渲染
- API 呼叫必須放在 `useEffect` 裡，否則每次渲染都觸發 → `setState` → 再次渲染 → 無限循環

### Hibernate 序列化
- `FetchType.LAZY` + Jackson 序列化會觸發 `ByteBuddyInterceptor` 錯誤
- 自我參照的 Entity 需要在 `@JsonIgnoreProperties` 加上自身欄位名稱防止無限遞迴

### 系統設計
- 潛在客戶分兩張表的理由：日常操作、資料量會累積、轉換流程需要乾淨的狀態管理
- `prospect_id` 存在 `customers`：保留開發履歷的查詢路徑

### Vite Proxy
- 本地開發時 `/api` 請求不會自動轉發後端
- `vite.config.ts` 的 `server.proxy` 設定只在本地開發有效
- 正式環境由 `vercel.json` 的 `rewrites` 處理轉發
