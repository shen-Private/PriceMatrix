## 技術債處理

### SalesQuoteItem 耦合修正 ✅

**問題：** `SalesQuoteItem` 直接引用 `pricing` 模組的 `Product` Entity（跨模組耦合）

**修正內容：**

#### DB
```sql
ALTER TABLE sales_quote_item
  ADD COLUMN product_name VARCHAR(255),
  ADD COLUMN product_code VARCHAR(100),
  ADD COLUMN base_price NUMERIC(38,2);
```

#### SalesQuoteItem.java
- 移除 `@ManyToOne Product product`
- 改為 `Long productId`（純 ID）
- 新增快照欄位：`productName` / `productCode` / `basePrice`
- 移除 `import Product`

#### SalesQuoteService.java
- `createQuote` / `createRevision` / `updateQuote` 三處改為寫入快照欄位
- `item.setProduct(product)` → `item.setProductId / setProductName / setBasePrice`
- 注意：`Product.getName()`（不是 `getProductName()`）

**學習重點：**
- 快照（Snapshot）= 建立時複製當下的值，之後與原始資料脫鉤
- 報價單是財務文件，商品改名/改價/下架不應影響歷史記錄
- `@ManyToOne` 存的是物件引用，Hibernate 取資料時自動 JOIN 查當下值

---

### SalesQuote Customer 耦合 ✅ 確認保留

**結論：** 不做快照，保留 `@ManyToOne Customer`

**理由：**
- 客戶資料修改代表現在的資訊更準確，報價單跟著更新是合理的
- 快照會造成歷史報價單與 `pricing_customers` 資料對不起來
- 客戶有 soft delete 保護，不會憑空消失

## 客戶停用/啟用功能

### 設計決策
- 停用是高風險操作 → 彈出 Modal，要求輸入客戶名稱確認
- 啟用風險較低 → `window.confirm` 簡單確認
- 停用後客戶仍顯示在列表（灰色樣式），可隨時啟用
- 操作權限：`deactivate_customer`（admin）

### 後端
- `CustomerService.java`：新增 `updateStatus(Long id, boolean isActive)`
- `CustomerController.java`：新增 `PATCH /{id}/status`，接收 `{ isActive: boolean }`
- 原有 `PATCH /{id}/deactivate` 保留（停用專用，不影響現有邏輯）

### 前端 AuthContext.tsx
- `Action` type 新增 `'deactivate_customer'`
- `admin` permissions 新增 `'deactivate_customer'`

### 前端 CustomerPanel.tsx
- `Customer` interface 新增 `active: boolean`
- `empty()` 新增 `active: true`
- 新增 state：`deactivateTarget` / `deactivateInput`
- 新增函式：`handleDeactivate`（停用，需輸入名稱確認）/ `handleActivate`（啟用，window.confirm）
- 列表按鈕：active → 紅色「停用」/ inactive → 藍色「啟用」，灰色文字顯示停用客戶
- 停用確認 Modal：顯示客戶名稱，輸入框驗證，錯誤提示
